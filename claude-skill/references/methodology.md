# Methodology

Derived by live experimentation against a real `codex-cli` 0.142.5 install, not documentation — Codex's docs don't cover backgrounding `exec` or its cost telemetry in the depth needed here.

## How each finding was derived

- **MCP `codex`/`codex-reply` are fully synchronous, no background primitives.** Called both tools directly (`mcp__codex__codex` then `mcp__codex__codex-reply` with the returned `threadId`) with a trivial prompt; each call blocked until a complete result came back. Inspected both tools' full JSON schemas directly — confirmed no `background`/`async` field or sibling list/stop/status tool exists anywhere in the MCP server's tool set.
- **No native background/session-manager CLI surface for `exec`.** Read `codex --help` and `codex exec --help` in full; the only session-management verbs (`resume`/`fork`/`archive`/`delete`/`unarchive`) are for *interactive* sessions, not `exec`. Confirmed `codex cloud` is the only genuinely async local-CLI-invoked mode, and it requires `--env` and creates cloud state — out of scope per explicit instruction to avoid cloud usage.
- **Backgrounding via the shell works, and `--json` gives clean, parseable stdout.** Ran `codex exec --json "<bounded loop prompt>" > out.jsonl 2> err.log &`, captured `$!`, and read the growing file — confirmed `thread.started` is the literal first line (usable as an immediate session-id source) and that `item.*`/`turn.completed` events are clean single-purpose JSON lines, not a rendered terminal buffer.
- **`turn.completed.usage` needs no dedup, unlike Claude Code's transcript.** Compared directly against the `claude-code-bg-sessions` skill's finding (Claude Code repeats a turn's full usage once per content-block line and must be deduped by `message.id`). Codex's stdout stream emits exactly one `turn.completed` per turn with the usage already finalized — confirmed by counting occurrences against the number of turns actually run.
- **Kill via plain `kill <pid>` is near-instant.** Same method as the Claude Code skill: captured the PID from `$!` at spawn time, timed the `kill` call's own wall-clock return, then polled `ps -p <pid>` at short intervals until it disappeared. Also confirmed the interruption is genuine (not "finish current turn then stop"): the killed run's stdout had no trailing `turn.completed`, and the on-disk rollout file for that thread stopped at the same point.
- **Resume genuinely continues context — but silently starts a new thread on a bad id.** First attempt passed an empty string as the thread id (a bug in the test harness, not the product) and got a fluent but *wrong* answer back from what turned out to be a brand-new thread — no error was raised. Re-ran with the correct thread id captured from the killed run's `thread.started` event and got the correct recalled answer, with the same `thread_id` echoed back. The contrast between these two runs is what surfaced the footgun — worth re-confirming (check the echoed `thread_id`) any time resume's answer looks suspicious.
- **Two distinct JSONL schemas exist (stdout stream vs. on-disk rollout file).** Noticed while trying to extract a session id from stdout using field names (`session_meta`) that turned out to only exist in the on-disk rollout format; the actual stdout schema uses `thread.started`/`item.*`/`turn.completed`. Confirmed both files exist for the same thread (matched by the shared UUID) and diffed their structures directly.
- **Approval/trust does not gate headless `exec` — tested directly, inference from the first pass was wrong.** The first pass only observed a `trusted` project and inferred (without testing) that an untrusted directory might hang `exec` waiting for an approval it can't surface headlessly. Later tested directly: created `/tmp/cool-project` fresh, started the interactive CLI there and declined the trust prompt ("No"), confirmed via `~/.codex/config.toml` that the directory has **no** project entry at all (declining doesn't persist an explicit `untrusted` record the way some other projects have one), then ran `codex exec --json -C /tmp/cool-project "run an echo command and report it"` against it directly. It executed the shell command and completed the full turn in ~11s — no hang, no approval prompt, behavior indistinguishable from the trusted-project case. Corrected the skill body to state this as confirmed, not inferred.

## What's most likely to have drifted if this misbehaves

1. Exact field names inside `turn.completed`/`token_count`/`item.*` payloads.
2. The kill-latency figures (a measurement, not a documented guarantee).
3. The specific stderr warning text observed (`Shell snapshot validation failed...`) — environment-specific noise, not a stable contract.

Least likely to have drifted: the general shape (`exec` as one-shot/backgroundable, `resume` by thread id, no native stop command, two JSONL schemas existing).
